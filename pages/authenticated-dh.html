Authenticated Diffie-Hellman

The MITM attack exploits the absence of identity verification in vanilla DH. The fix is straightforward: digitally sign your public key with a long-term identity key. Eve cannot forge a valid signature — her interception attempt is detected immediately.

Unauthenticated DH (vulnerable)

  • ✗ No identity binding — anyone can pose as Alice or Bob
  • ✗ Eve intercepts Y_A, substitutes Y_E — undetected
  • ✗ Two separate shared secrets created
  • ✗ Eve reads and re-encrypts all messages
  • ✗ No way to detect the attack at protocol level

Authenticated DH (secure)

  • ✓ Each party signs their DH public key with their identity key
  • ✓ Eve cannot produce a valid signature for Alice's identity
  • ✓ Recipient verifies signature before accepting the public key
  • ✓ Eve's substituted key fails verification — alert raised
  • ✓ Used in TLS 1.3, Signal Protocol, SSH

Step-by-Step Breakdown

1

Long-Term Identity Keys

Before any session, Alice and Bob each have a long-term identity key pair whose public components are certified by a trusted authority (CA or out-of-band verification). These are separate from the session DH keys.

2

Sign the DH Public Key

Alice sends (Y_A, Sig_A) where Sig_A = Sign(sk_A, Y_A). Bob sends (Y_B, Sig_B). The signature cryptographically binds the DH public key to the sender's verified identity.

3

Eve Tries to Intercept

Eve intercepts Alice's message and substitutes her own (Y_E, Sig_E). Bob attempts to verify: Verify(pk_A, Y_E, Sig_E) — this fails because Sig_E was created with Eve's key, not Alice's. Bob rejects the message and aborts.

4

Legitimate Exchange Succeeds

With Eve blocked, Alice and Bob complete the exchange using their authentic (Y_A, Y_B) values, arriving at a shared secret K that only they can compute. The MITM attack is defeated.
